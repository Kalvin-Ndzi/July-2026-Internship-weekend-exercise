# Pre-CSS Checkpoint

A set of exercises to run between "students can write clean HTML" and "let's add CSS." The goal is to test understanding, not just pattern-copying, before a new layer (styling) makes structural mistakes easier to hide visually.

## Suggested order

| # | Folder | What it tests | Format |
|---|--------|---------------|--------|
| 1 | `01-debugging-exercises` | Can they *recognize* broken semantics, not just produce correct ones from memory? | 3 broken HTML files + answer key |
| 2 | `02-blueprint-to-html` | Can they choose the right tags themselves with no starter skeleton? | 3 written briefs, no HTML provided |
| 3 | `03-diagnostic-quiz` | Quick surface-check of conceptual gaps before CSS piles on top | 10 short questions + answer key |
| 4 | `04-peer-review-checklist` | Can they explain *why* something is wrong in someone else's code? | Checklist for swapping files |
| 5 | `05-validator-pass` | Installing a lifelong habit: validate before you style | Short guide, no answer key needed |
| 6 | `06-capstone-project` | Final checkpoint — everything together, one real page | Brief + rubric |

## How I'd run this across a session or two

1. **Debugging exercises** first, as a warm-up — low stakes, gets them actively hunting for problems.
2. **Diagnostic quiz** right after — quick, surfaces anything the debugging exercise didn't catch.
3. **Blueprint-to-HTML** as the main event — this is the real test of independent tag choice.
4. **Peer review** using each other's blueprint pages from step 3 — reuses their own work, adds a second pair of eyes.
5. **Validator pass** on whatever they've built so far — 10 minutes, teach the habit.
6. **Capstone** as the "you're ready for CSS" gate — assign it, and don't move to CSS until most of the class clears it.

Nothing here uses CSS. That's deliberate — the point is to confirm the structural foundation is solid before a new layer makes it easy to paper over a missing `<label>` or a skipped heading level with something that just *looks* fine.
